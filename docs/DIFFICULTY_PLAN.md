# 난이도 상향 계획 — 함정 정의와 실측 기반 재설계

작성: 2026-07-20 · 근거: `assets/campaign/bank.json` 775판 전수 시뮬레이션
전제: 모든 수치는 **출하된 bank.json을 그대로 돌려 측정한 값**이다.
(기존 `PUZZLE_BANK_STRATEGY.md`의 audit 수치는 `campaign.json`에서 파라미터를 바꿔
**즉석 생성한 보드**를 재서 실물과 다르다. 이 문서는 실물을 잰다.)

---

## 0. "함정"의 정의 — 코드 기준

이 게임에서 실수는 **막힌 화살을 탭하는 것** 하나뿐이다. 막힘에는 성격이 다른 세 종류가
있고, 돈이 되는 건 뒤 두 개다.

| 종류 | 판정(`board_logic.dart`) | 플레이어 | 현재 비율 |
|---|---|---|---|
| **벽** | `freeSteps = 0` — 화살촉 바로 앞이 막힘 | 눈에 보임 → **실수 안 함** | **74.9%** |
| **공간 함정** | `freeSteps ≥ 3` — 광선이 여러 칸 뻗다 **멀리서** 막힘 | "아, 아까워"(내 탓) | **9.3%** |
| **시간 함정** | 지금 막혔지만 **다른 화살을 먼저 빼면** 나감 (순서) | 순서 안 짜면 실수 | 사실상 **0** |

- **공간 함정** = `tool/audit_traps.dart`가 재는 "막힘 거리". 광선이 길수록 멀리서
  막힐 여지가 생긴다 → 사용자 지적 #1·#2·#3과 직결.
- **시간 함정** = `tool/audit_diff.dart`의 **분기수(branching)** = "매 순간 뺄 수 있는
  화살 개수". 낮을수록 순서를 강제한다 → 사용자 지적 #4 "좌측이 빠져야 우측이 나감".

> **난이도 = 벽을 줄이고, 공간 함정 + 시간 함정을 늘리는 것.**

## 1. 실측한 현재 상태 (bank.json 775판)

| 지표 | 현재 | 목표 | 대응 지적 |
|---|---|---|---|
| 밀도(occupied/mask) | **97.1%** (요청 92% 무시됨) | 82~90% | 근본 원인 |
| 함정비율(3칸+ 막힘) | **9.3%** | 15~25% | — |
| **2셀 화살 비율** | **40.8%** (최대 61%) | ≤20% | **#2** |
| 롱(7셀+) 비율 | **13.6%** | ↑ | **#1** |
| 숏화살 뭉치(3화살+ 인접) | **45.4%** | ↓ | **#2** |
| **분기수(branching)** | **19.2** (최소 11) | 후반 3~5 | **#4** |
| 길이 구성(숏≤2 / 미들3-6 / 롱7+) | 42 / 44 / 14 % | 롱↑·2셀↓ | **#1·#3** |

**한 문장 진단:** 리페어 패스가 판을 **97%까지 꽉 채워** → 광선이 짧아지고 →
스텁(2셀)이 40% 양산되고 → 모든 화살이 서로 안 막혀서(분기 19) **아무 순서로나 풀린다.**
개수(80→300)만 맞추느라 정작 난이도를 만드는 세 지표(함정·구성·심도)는 아무도 안 본다.

## 2. 근본 원인 — `level_generator.dart`

1. **리페어 패스가 밀도 노브를 덮어쓴다.** 본 루프는 `fill`(0.92)까지만 채우지만,
   이후 (a)스텁·(b)스플라이스·(c)꼬리성장이 **남은 빈칸을 전부** 메워 97%가 된다.
   빽빽 → 광선 짧음 → 공간 함정 소멸.
2. **스텁/꼬리성장이 2셀 화살을 양산한다.** 리페어로 생기는 화살은 대부분 2셀.
3. **머리 방향을 가장자리 끝으로 먼저 잡는다.** 풀림 보장(역순 제거 불변식)을 만드는
   규칙이 동시에 **바깥 편향 82%**를 만들어, 초반이 무저항 구간이 된다.
4. **`export_boards.dart`는 개수만 게이트한다.** 길이 구성·함정·분기수는 통과 조건에 없다.

## 3. 계획 — 4개 워크스트림 (사용자 지적에 1:1 대응)

전 과정이 **오프라인 재생성**이다:
`tool/export_boards.dart` → `tools/atlas/all_boards.json` → `tools/atlas/build_bank.py`
→ `assets/campaign/bank.json` → 앱이 읽음. **런타임 코드·빌드·배포 변경 없음.**
상시 승인된 `dart analyze`·`flutter test`·재생성·커밋 범위 안에서 끝난다.

### WS1 — 밀도 노브 복구 + 2셀 스팸 제거 (선행 필수 · #2 근본)
- 리페어 패스가 목표 `fill`을 넘기면 **중단**하도록 상한을 건다. (지금은 무조건 다 메움)
- 보드당 **2셀 화살 상한(≤20%)**: 남는 스텁은 인접 화살 꼬리로 흡수하거나 빈칸으로 남긴다.
- 게이트: 밀도·2셀비율·뭉치율 회귀 측정.
- ⚠️ WS1이 안 되면 아래가 전부 무력화된다. **첫 착수 지점.**

### WS2 — 길이 "구성"을 목표로 (개수 대신 · #1·#3)
- `LenMixes`를 롱 쪽으로 재조정(+ `maxLen 20` 유지).
- `export_boards.dart`의 통과 게이트를 **"개수 밴드"에서 "길이 히스토그램 목표
  (숏/미들/롱 비율 + 분산)"로 교체.** 개수는 부산물이 된다.
- WS1로 밀도가 내려가면 광선이 길어져 **공간 함정이 자동으로 증가**한다.

### WS3 — 의존성 심도를 생성기 목표로 (#4 · 가장 깊은 변경)
- 머리 방향 선택을 무작위 쪽으로 옮겨 바깥 편향을 낮추고, 화살을 **서로의 광선 위에
  의도적으로 얹어** 순서를 강제한다.
- **역순 제거 불변식은 유지** → 항상 최소 1개는 free → **절대 데드락 없음(판이 안 죽음).**
  분기수만 19 → 목표대로 낮춘다.
- 게이트: `audit_diff` 분기수.
- ⚠️ 순서 강제는 "막는 화살이 화면 밖"이면 나쁜 실수가 된다. HANDOFF의 **판 크기 상한
  결정**과 함께 가야 한다(거대 국가 보드).

### WS4 — 곡선 + 회귀 테스트
- audit 3종(`traps`·`mistakes`·`diff`)을 **실제 bank.json 대상**으로 바꿔 회귀 테스트로
  승격한다(지금은 실물을 안 본다).
- 스테이지별 목표 곡선: 초반 라운드는 벽만(실수 0) → 후반으로 갈수록 함정·심도 ↑.

**착수 순서:** WS1(밀도) → WS2(구성) → WS3(심도) → WS4(고정).

## 4. 착수 전 확정할 결정 2건

> 아래 두 갈래는 커밋된 스펙과 충돌하거나(‑> #3) 수익/이탈을 가르는 튜닝 지점이라,
> 실제 코드 변경 **전에** 사용자 확정이 필요하다. 괄호는 담당 추천안.

### 결정 A — 난이도 목표 기준 (⚠️ 커밋된 스펙과 충돌)
`export_boards.dart` 상단 확정 규격은 **화살 개수(80→300 램프)**를 난이도로 삼는데,
사용자 지적 #3은 "개수가 아니라 구성"이다. 셋 중 택1:
- **(추천) 개수 스펙 폐기 → 구성+심도를 목표로.** 개수는 부산물.
  단점: '후반=큰 판' 웅장함 램프가 약해진다.
- 개수 유지, 그 안에서 구성만 조정. (보수적 · 밀도 압박 잔존)
- 혼합: 라운드 후반(보스)만 개수 유지, 초·중반은 구성+심도.

### 결정 B — 난이도 강도 (실수/분기 곡선)
- **(추천) `PUZZLE_BANK_STRATEGY.md` 4장 곡선:** 초반 실수 0(온보딩), 31라운드+ 판당
  1~2, 라운드끝 2~3. 분기수 19 → 후반 3~5. '절대 안 죽는 판' 유지.
- 더 공격적(수익 우선): 소진율↑, 이탈 위험↑.
- 일단 만들고 실측 곡선을 보여준 뒤 함께 튜닝.
